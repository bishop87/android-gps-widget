# Change Request — CR-002  
**Titolo:** Estensione funzionalità configurazione API GPA e schermata Mappa  
**Data:** 2026-02-28  
**Richiedente:** Product / Tech  
**Tipo:** Functional Enhancement  
**Priorità:** Media-Alta  
**Stato:** Proposed  

---

## 1. Scopo della Change Request

Questa Change Request introduce:

- configurazione endpoint API GPA nelle Impostazioni  
- accesso rapido alla schermata Mappa dalla Home  
- nuova schermata Mappa con visualizzazione tracciati GPS filtrabili per intervallo temporale  

La presente CR **integra** i requisiti esistenti senza modificarli.

---

## 2. Ambito

### In scope

- UI Impostazioni (nuovo campo)
- UI Home (nuovo bottone)
- Nuova schermata Mappa
- Integrazione Google Maps
- Fetch dati GPS da API configurabile
- Visualizzazione marker e tracciato
- Filtri temporali
- Posizione corrente dispositivo

### Out of scope

- Modifiche al formato API backend
- Funzionalità di editing dei punti GPS
- Caching offline avanzato

---

## 3. Requisiti Funzionali

---

### RF-SET-API-01 — Configurazione endpoint GPA

**Descrizione**  
L’applicazione deve consentire all’utente di configurare l’endpoint API per il recupero dei dati GPA.

**Dettagli**

- Nella schermata **Impostazioni** deve essere aggiunto:
  - un campo di input testuale editabile
- Il campo deve permettere l’inserimento dell’URL dell’API di fetch dei dati GPA.
- Il valore inserito deve essere:
  - validato come URL valido
  - salvato in modo persistente nelle preferenze dell’app
  - riutilizzato per le chiamate di fetch dati
- Deve essere previsto un valore di default configurabile.

**Acceptance Criteria**

- Dato un URL valido → viene salvato correttamente  
- Dato un URL non valido → viene mostrato errore  
- Al riavvio app → il valore persiste  

---

### RF-HOME-MAP-01 — Accesso schermata mappa

**Descrizione**  
L’applicazione deve fornire accesso rapido alla schermata mappa dalla home.

**Dettagli**

- Nella schermata **Home**
- Posizione: sotto al bottone **Test API**
- Deve essere presente un bottone con etichetta:

> **"Mappa"**

**Comportamento**

- Al tap l’app deve aprire la **Schermata Mappa**.

**Acceptance Criteria**

- Il bottone è visibile in Home  
- Il tap apre la schermata corretta  
- Nessun crash in assenza di configurazione API  

---

### RF-MAP-UI-01 — Visualizzazione mappa

**Descrizione**  
La schermata mappa deve mostrare una mappa interattiva a pieno schermo.

**Dettagli**

- Deve essere utilizzato **Google Maps SDK**
- La mappa deve occupare tutto lo spazio disponibile della schermata.
- Deve essere presente il controllo per il cambio layer mappa.

**Acceptance Criteria**

- La mappa viene caricata correttamente  
- Lo zoom e pan funzionano  
- Il cambio layer è operativo  

---

### RF-MAP-FILTER-01 — Filtri temporali

**Descrizione**  
L’utente deve poter filtrare i dati GPS per intervallo temporale.

**Elementi UI (sotto la mappa)**

- DateTime Picker **Data/Ora Inizio**
- DateTime Picker **Data/Ora Fine**

**Comportamento**

- I valori selezionati devono essere utilizzati come filtri nella chiamata API.
- Deve essere impedita la selezione di un intervallo non valido (fine < inizio).

**Acceptance Criteria**

- Non è possibile confermare intervalli invalidi  
- I filtri vengono passati alla API  
- I filtri persistono durante la sessione della schermata  

---

### RF-MAP-DATA-01 — Fetch dati GPS

**Descrizione**  
L’app deve scaricare i punti GPS dall’API configurata.

**Comportamento**

- La chiamata API deve:
  - usare l’endpoint configurato nelle impostazioni
  - includere i filtri data/ora selezionati
- I dati ricevuti devono essere visualizzati sulla mappa.

**Acceptance Criteria**

- Con endpoint valido → dati caricati  
- Con errore API → messaggio utente  
- Con dataset vuoto → stato vuoto gestito  

---

### RF-MAP-VIS-01 — Visualizzazione marker

**Descrizione**  
I punti GPS devono essere rappresentati graficamente sulla mappa.

**Requisiti**

- Per ogni punto GPS deve essere mostrato:
  - marker piccolo
  - forma rotonda
- I marker devono essere collegati tra loro in ordine temporale tramite:
  - linea continua (polyline)
- L’ultimo marker in ordine temporale deve:
  - avere colore differente
  - essere visivamente distinguibile.

**Acceptance Criteria**

- Numero marker = numero punti  
- Polyline segue ordine temporale  
- Ultimo punto chiaramente distinguibile  

---

### RF-MAP-DETAIL-01 — Dettaglio marker

**Descrizione**  
L’utente deve poter visualizzare i dettagli di un punto GPS.

**Comportamento**

- Al tap su un marker deve essere mostrato un info window con:
  - timestamp
  - coordinate
  - eventuali dati aggiuntivi disponibili dall’API.

**Acceptance Criteria**

- Tap su marker → info visibile  
- Dati coerenti con API  
- Info window chiudibile  

---

### RF-MAP-POS-01 — Posizione corrente

**Descrizione**  
La mappa deve mostrare la posizione corrente del dispositivo.

**Requisiti**

- Deve essere richiesto il permesso di localizzazione.
- Deve essere visualizzato il marker di posizione corrente.
- Deve essere disponibile il pulsante standard di centratura.

**Acceptance Criteria**

- Con permesso concesso → posizione visibile  
- Con permesso negato → gestione graceful  
- Pulsante centratura funzionante

## 3. Specifica integrazione API

### Endpoint

L’app deve interrogare un endpoint configurabile compatibile con:


GET /map_data2.php


### Autenticazione

La chiamata all'API deve includere l'autenticazione tramite **Basic Auth**, utilizzando le stesse credenziali (username e password) già in uso e salvate nelle impostazioni correnti dell'applicazione.


### Parametri supportati

| Parametro | Tipo | Obbligatorio | Note |
|----------|------|--------------|------|
| device | string | no | se omesso → tutti i device |
| date_from | string | no | formato `Y-m-d H:i` |
| date_to | string | no | formato `Y-m-d H:i` |

### Formato date richiesto

**ATTENZIONE (vincolo forte)**  
Il backend accetta esclusivamente:


YYYY-MM-DD HH:mm


Esempio valido:


2026-02-28 14:30


L’app **NON deve inviare i secondi**.
